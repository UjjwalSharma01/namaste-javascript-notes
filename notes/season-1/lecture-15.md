## ‚ö° JavaScript's Asynchronous Foundation

> **üîë Key Truth:** Call stack will execute any execution context which enters it. Time, tide and JS waits for none. **TLDR: Call stack has no timer.**

## üåê Web APIs: Browser's Superpowers

### üìö **What are Web APIs?**

**None of the below are part of JavaScript!** These are extra superpowers that browser provides. Browser gives access to JS call stack to use these powers.

![Event Loop 2 Demo](/assets/eventloop2.jpg)

### üîß **Common Web APIs:**

| Web API | Purpose | Example Usage |
|---------|---------|---------------|
| **setTimeout()** | Timer function | `setTimeout(callback, delay)` |
| **DOM APIs** | HTML DOM tree access | `document.getElementById()` |
| **fetch()** | Network requests | `fetch('https://api.example.com')` |
| **localStorage** | Browser storage | `localStorage.setItem()` |
| **console** | Developer tools | `console.log()` (Yes, even console is not JS!) |
| **location** | URL/navigation | `window.location.href` |

### ü™ü **Window Object: The Global Gateway**

We get all these APIs through the **global `window` object**:

```javascript
// These are all equivalent:
setTimeout(callback, 1000);
window.setTimeout(callback, 1000);

console.log("Hello");
window.console.log("Hello");

localStorage.setItem("key", "value");
window.localStorage.setItem("key", "value");
```

**üí° Why we don't write `window`:** As `window` is the global object, and all Web APIs are present in it, we don't explicitly write `window` - but it's always implied.

## üîÑ Event Loop in Action

### üîç **Basic Example: Understanding the Flow**

![Event Loop 3 Demo](/assets/eventloop3.jpg)

```javascript
console.log("start");
setTimeout(function cb() {
  console.log("timer");
}, 5000);
console.log("end");

// Output:
// start
// end
// timer (after 5 seconds)
```

### üß† **Step-by-Step Execution:**

#### **Phase 1: Synchronous Execution**
1. **GEC created** ‚Üí Global Execution Context enters call stack
2. **`console.log("start")`** ‚Üí Calls console Web API ‚Üí Prints "start"
3. **`setTimeout(..., 5000)`** ‚Üí Calls setTimeout Web API:
   - Stores callback `cb()` in Web API environment
   - Starts 5-second timer
   - Returns immediately (non-blocking)
4. **`console.log("end")`** ‚Üí Calls console Web API ‚Üí Prints "end"
5. **GEC completes** ‚Üí Pops from call stack

#### **Phase 2: Timer Completion**
6. **Timer expires** ‚Üí After 5 seconds, `cb()` needs to execute
7. **But wait!** ‚Üí `cb()` cannot directly enter call stack
8. **Callback Queue** ‚Üí `cb()` first goes to callback queue
9. **Event Loop** ‚Üí Checks if call stack is empty, then moves `cb()` to call stack
10. **Execution** ‚Üí `cb()` executes, prints "timer", then pops from call stack

### üõ°Ô∏è **Event Loop: The Gatekeeper**

**Event Loop's Job:**
- Continuously monitor call stack and callback queue
- **If call stack is empty** AND **callback queue has tasks** ‚Üí Move task to call stack
- **Never interrupt** ongoing execution in call stack

## üìã Callback Queue Deep Dive

### ‚ùì **Q: How does the timer callback reach the call stack after 5 seconds?**

![Event Loop 4 Demo](/assets/eventloop4.jpg)

**Answer:** Through the **Callback Queue** and **Event Loop** mechanism:

1. **Timer expires** ‚Üí `cb()` ready for execution
2. **Enters callback queue** ‚Üí Waits for its turn
3. **Event loop monitors** ‚Üí Checks call stack status
4. **Call stack empty?** ‚Üí Event loop moves `cb()` to call stack
5. **Execution** ‚Üí `cb()` runs and completes

### üîç **Event Listener Example**

![Event Loop 5 Demo](/assets/eventloop5.jpg)

```javascript
console.log("Start");

document.getElementById("btn").addEventListener("click", function cb() {
  // cb() registered in Web API environment with click event attached
  console.log("Callback");
});

console.log("End");

// Output immediately:
// Start
// End
// 
// Output when button clicked:
// Callback
```

### üß† **What Happens:**

1. **Event listener registered** ‚Üí `cb()` stored in Web API environment
2. **Event attached** ‚Üí Browser watches for click events on button
3. **Execution continues** ‚Üí "Start" and "End" printed, GEC pops
4. **Event listener persists** ‚Üí Stays in Web API environment indefinitely
5. **User clicks** ‚Üí `cb()` moves to callback queue
6. **Event loop** ‚Üí Moves `cb()` to call stack for execution

#### **üö® Important:** Event listeners stay in Web API environment until explicitly removed or browser closed!

### ‚ùì **Q: Why do we need a callback queue?**

**Scenario:** User clicks button 6 times rapidly

**Answer:**
1. **6 callbacks** ‚Üí All 6 `cb()` instances enter callback queue
2. **Sequential processing** ‚Üí Event loop processes them one by one
3. **No conflicts** ‚Üí Each callback executes completely before next one
4. **FIFO order** ‚Üí First click processed first, last click processed last

**üí° Benefit:** Maintains order and prevents callback conflicts.

## ‚ö° Microtask Queue: The Priority Lane

### üîç **Fetch API Behavior**

```javascript
console.log("Start");

setTimeout(function cbT() {
  console.log("CB Timeout");
}, 5000);

fetch("https://api.netflix.com").then(function cbF() {
  console.log("CB Netflix");
}); // Takes 2 seconds to get response

// Millions of lines of code here...

console.log("End");

// Actual Output:
// Start
// End
// CB Netflix (after 2 seconds)
// CB Timeout (after 5 seconds)
```

### üß† **Execution Analysis:**

#### **Phase 1: Registration**
1. **`console.log("Start")`** ‚Üí Prints "Start"
2. **`setTimeout`** ‚Üí `cbT` registered in Web API, 5-second timer starts
3. **`fetch`** ‚Üí `cbF` registered in Web API, network request initiated
4. **Millions of lines** ‚Üí Execute synchronously
5. **`console.log("End")`** ‚Üí Prints "End"

#### **Phase 2: Async Completion**
6. **After 2 seconds** ‚Üí Netflix response ready, `cbF` enters **Microtask Queue**
7. **After 5 seconds** ‚Üí Timer expires, `cbT` enters **Callback Queue**
8. **Event loop priority** ‚Üí Processes Microtask Queue first
9. **`cbF` executes** ‚Üí Prints "CB Netflix"
10. **`cbT` executes** ‚Üí Prints "CB Timeout"

![Event Loop 6 Demo](/assets/eventloop6.jpg)

### üèÜ **Priority System**

![Microtask Priority Visualization](/assets/microtask.gif)

```
Event Loop Priority:
1. Call Stack (highest priority)
2. Microtask Queue 
3. Callback Queue (lowest priority)
```

#### **‚ö° Microtask Queue gets higher priority than Callback Queue!**

### üìä **What Goes Where?**

| Queue Type | Contents | Examples |
|------------|----------|----------|
| **Microtask Queue** | Promise callbacks, Mutation Observer | `.then()`, `.catch()`, `.finally()` |
| **Callback Queue** | All other async callbacks | `setTimeout`, `setInterval`, DOM events |

#### **üîç Detailed Breakdown:**

**Microtask Queue:**
- **Promise callbacks** ‚Üí `.then()`, `.catch()`, `.finally()`
- **Mutation Observer** ‚Üí Watches DOM changes and executes callbacks
- **queueMicrotask()** ‚Üí Manually queued microtasks

**Callback Queue (Task Queue):**
- **Timer callbacks** ‚Üí `setTimeout`, `setInterval`
- **DOM events** ‚Üí Click, scroll, resize handlers
- **I/O operations** ‚Üí File operations, network (excluding fetch promises)

### ‚ö†Ô∏è **Starvation Problem**

```javascript
// ‚ùå This can cause starvation
function recursiveMicrotask() {
  Promise.resolve().then(() => {
    console.log("Microtask");
    recursiveMicrotask(); // Creates new microtask
  });
}

setTimeout(() => {
  console.log("This may never run!"); // Starved callback
}, 0);

recursiveMicrotask();
```

**Problem:** If microtask queue keeps creating new tasks, callback queue never gets a chance to run!

## ‚ùì Important Questions & Answers

### **1. When does the event loop actually start?**

**Answer:** Event loop, as the name suggests, is a single-thread, loop that is **almost infinite**. It's **always running** and doing its job from the moment the JavaScript runtime starts.

```javascript
// Event loop is running even before this code executes
console.log("Event loop was already running!");
```

### **2. Are only asynchronous Web API callbacks registered in Web API environment?**

**Answer:** **YES**, only asynchronous callbacks go through Web APIs. Synchronous callbacks like those in `map`, `filter`, and `reduce` are **not** registered in Web API environment.

```javascript
// ‚ùå NOT in Web API (synchronous callback)
[1, 2, 3].map(function(x) {
  return x * 2;
});

// ‚úÖ IN Web API (asynchronous callback)
setTimeout(function() {
  console.log("Async callback");
}, 1000);
```

### **3. Does Web API environment store the callback function and push the same to queue?**

**Answer:** **Yes**, callback functions are stored in Web API environment, and a **reference** is scheduled in the queues. 

**‚ö†Ô∏è Special Case:** Event listeners (click handlers) stay in Web API environment **forever** until explicitly removed. This is why it's advised to remove listeners when not needed for garbage collection.

```javascript
// Event listener stays in Web API forever
button.addEventListener('click', handleClick);

// Good practice: Remove when done
button.removeEventListener('click', handleClick);
```

### **4. What if setTimeout delay is 0ms?**

**Answer:** There are **trust issues** with `setTimeout()` üòÖ. Even with 0ms delay:

```javascript
setTimeout(() => {
  console.log("This might wait!");
}, 0);

// Heavy synchronous operation
for (let i = 0; i < 1000000000; i++) {
  // Blocking operation
}

console.log("This runs first!");
```

**Why?** The callback needs to wait until the **call stack is empty**. So the 0ms callback might wait for 100ms+ if the stack is busy.

## üìã Quick Summary

### üí° What We Learned:

#### **1. Browser Architecture**
- JavaScript Engine + Web APIs + Event Loop System
- Web APIs provide async capabilities to single-threaded JavaScript
- Window object is the gateway to all Web APIs

#### **2. Event Loop Mechanism**
- Call Stack executes synchronous code
- Web APIs handle asynchronous operations
- Event Loop coordinates between queues and call stack

#### **3. Queue Priority System**
- **Microtask Queue** ‚Üí Higher priority (Promises, Mutation Observer)
- **Callback Queue** ‚Üí Lower priority (setTimeout, DOM events)
- Event Loop always processes microtasks before callbacks

#### **4. Common Patterns**
- Timer callbacks go through callback queue
- Promise callbacks go through microtask queue
- Event listeners persist in Web API environment

### üß† Quick Memory Aid:
```
Single Thread + Web APIs = Async JavaScript
Call Stack ‚Üí Microtask Queue ‚Üí Callback Queue
Promises = Microtask (high priority)
setTimeout = Callback (low priority)  
Event Loop = The coordinator between all
```
